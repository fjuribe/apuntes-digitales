🏴‍☠️ Caso CTF: Banner Grabbing

En un CTF o pentest, lo primero que haces con netcat es banner grabbing: conectarte a puertos abiertos para ver qué servicio y versión está corriendo. Con esa versión buscas en searchsploit o exploit-db.com si tiene vulnerabilidades conocidas (CVEs).

FTP — File Transfer Protocol

Puerto 21

Comandos FTP en sesión interactiva

bash — sesión FTP
# Conectar al servidor
ftp 10.0.1.6

# Dentro de la sesión FTP:
ftp> ls           # listar archivos
ftp> get file.bin  # descargar archivo
ftp> put local.txt  # subir archivo
ftp> cd /directorio # cambiar directorio
ftp> mget *.txt     # descargar múltiples
ftp> pwd           # directorio actual
ftp> passive       # modo pasivo (bypass NAT/firewall)
ftp> binary        # modo binario (para archivos no-texto)
ftp> quit          # cerrar sesión

# Login anónimo (pentesting)
ftp 10.0.1.6
# user: anonymous | pass: (enter vacío o email falso)
⚠️ Vulnerabilidad crítica

FTP transmite TODO en texto plano: usuario, contraseña, y datos. Cualquier atacante en la misma red puede capturarlo con tcpdump/Wireshark. En producción usa SFTP (SSH, puerto 22) o FTPS (FTP + TLS, puerto 990).

🏴‍☠️ Técnica: Capturar credenciales FTP

Si estás en la misma red que el servidor FTP y logras hacer ARP Spoofing (ver sección ARP), puedes interceptar la sesión:

bash
# Capturar credenciales FTP en la red
sudo tcpdump -i eth0 port 21 -A | grep -E "USER|PASS"

# En Wireshark, filtro específico para credenciales:
ftp.request.command == "USER" or ftp.request.command == "PASS"
📋 Escenario de Práctica

Lab: Usa una VM con vsftpd o proftpd habilitado con anonymous login. Conéctate desde otra VM y usa Wireshark para capturar tus propias credenciales en texto plano. Observa cómo el paquete muestra USER admin y PASS contraseña123 en texto claro.


Defensa: Configura el servidor para aceptar solo FTPS o migra a SFTP. Compara los paquetes capturados — con TLS/SSH no se ve nada legible.

🔬 Para Profundizar
  • Diferencia entre modo activo vs. pasivo en FTP (importante con firewalls)
  • Herramienta hydra para fuerza bruta en FTP: hydra -l admin -P wordlist.txt ftp://10.0.1.6
  • Configurar VSFTPD en Linux con TLS habilitado
  • Anonymous FTP en CTFs: siempre probar anonymous:anonymous

DHCP — Asignación Dinámica de IPs

UDP 67/68

Flujo DORA — cómo obtienes tu IP

Discover
→
Offer
→
Request
→
ACK

Discover: el cliente broadcast "¿hay algún servidor DHCP?" → Offer: el servidor responde con una IP disponible → Request: el cliente acepta la oferta → ACK: el servidor confirma la asignación.

🔴 Ataque: DHCP Starvation▶

¿Qué es? El atacante envía miles de solicitudes DHCP con MACs falsas, agotando el pool de IPs disponibles. Los clientes legítimos ya no pueden obtener IP → denegación de servicio.


bash — herramienta yersinia
# DHCP Starvation con yersinia
sudo yersinia dhcp -attack 1 -interface eth0

# O con dhcpstarv:
sudo dhcpstarv -i eth0

Defensa — DHCP Snooping en switches: El switch crea una tabla de confianza. Solo los puertos "trusted" (uplinks) pueden enviar DHCP Offers. Los puertos de usuarios solo pueden hacer Discover/Request, nunca Offer/ACK.

cisco ios
! Habilitar DHCP Snooping globalmente
Switch(config)# ip dhcp snooping
Switch(config)# ip dhcp snooping vlan 10

! Marcar el puerto del servidor DHCP como "trusted"
Switch(config-if)# ip dhcp snooping trust

! Limitar solicitudes por segundo en puerto usuario
Switch(config-if)# ip dhcp snooping limit rate 10
🔴 Ataque: Rogue DHCP Server▶

Un atacante levanta su propio servidor DHCP. Si responde primero al Discover del cliente, le asigna una IP con el gateway del atacante como default gateway. Todo el tráfico del cliente pasa por el atacante → MitM sin ARP Spoofing.


python — rogue DHCP con scapy
# Ejemplo conceptual con scapy
from scapy.all import *

def rogue_dhcp(pkt):
    if DHCP in pkt and pkt[DHCP].options[0][1] == 1: # Discover
        # Construir DHCP Offer malicioso
        # gateway = IP del atacante
        pass

sniff(filter="udp and port 67", prn=rogue_dhcp)

Defensa: DHCP Snooping previene esto al bloquear Offers desde puertos no confiables.

🔬 Para Profundizar
  • Configurar servidor DHCP en Linux: sudo apt install isc-dhcp-server, archivo /etc/dhcp/dhcpd.conf
  • Room TryHackMe: "Networking Fundamentals" → módulo DHCP
  • Investigar: DHCP Option 82 (información de relay) para trazabilidad

ARP Spoofing / Poisoning

Capa 2

¿Cómo funciona ARP?

ARP (Address Resolution Protocol) mapea IPs a MACs dentro de la red local. Cuando quieres enviar datos a 192.168.1.1, tu PC broadcast: "¿Quién tiene 192.168.1.1? Dime tu MAC." El dueño de esa IP responde con su MAC.

bash — ver tabla ARP
# Ver tu tabla ARP local (IPs ↔ MACs conocidas)
arp -a      # Linux/Mac/Windows

# Ejemplo de salida:
# ? (192.168.1.1) at aa:bb:cc:dd:ee:ff [ether] on eth0
# ? (192.168.1.20) at 11:22:33:44:55:66 [ether] on eth0

ARP Spoofing — el ataque MitM clásico

El atacante envía ARP Replies falsos. Le dice a la víctima "yo soy el gateway" y le dice al gateway "yo soy la víctima". Ambos actualizan sus tablas ARP → todo el tráfico pasa por el atacante.

Víctima
→
Atacante
→
Gateway
El atacante reenvía (y puede modificar) el tráfico entre víctima y gateway
🏴‍☠️ Herramientas de ARP Spoofing
bash
# arpspoof (dsniff package)
# Envenenar a la víctima diciéndole que el atacante es el gateway
sudo arpspoof -i eth0 -t VICTIMA_IP GATEWAY_IP

# Y al gateway diciéndole que el atacante es la víctima
sudo arpspoof -i eth0 -t GATEWAY_IP VICTIMA_IP

# Habilitar IP forwarding para no cortar el tráfico
echo 1 > /proc/sys/net/ipv4/ip_forward

# bettercap (más moderno y completo)
sudo bettercap -iface eth0
# bettercap prompt:
net.probe on
arp.spoof on
net.sniff on
🛡 Defensa: Dynamic ARP Inspection (DAI)▶

DAI trabaja con DHCP Snooping: usa la tabla DHCP Snooping Binding (IP ↔ MAC ↔ Puerto) para validar cada ARP Reply. Si la MAC-IP del Reply no coincide con la binding table → el paquete es descartado.

cisco ios
Switch(config)# ip arp inspection vlan 10

! Puertos trusted (uplinks) no validan
Switch(config-if)# ip arp inspection trust

! Ver violaciones
Switch# show ip arp inspection statistics
📋 Lab: ARP Spoofing en red virtual
  1. Crea 3 VMs: atacante (Kali), víctima (Ubuntu), gateway (cualquiera)
  2. En Kali: corre arp -a para ver la tabla inicial
  3. Ejecuta arpspoof bidireccional + habilita ip_forward
  4. En Wireshark del atacante: observa el tráfico HTTP de la víctima
  5. En la víctima: ejecuta arp -a y nota cómo la MAC del gateway cambió
  6. Activa DAI en un switch virtual (GNS3/Packet Tracer) y repite — el ataque no funciona

ICMP — Diagnóstico de Red

Capa 3

ping y traceroute

bash
# ping — verifica conectividad y mide latencia
ping 8.8.8.8              # ping continuo
ping -c 4 8.8.8.8        # 4 paquetes y para
ping -i 0.2 8.8.8.8      # ping rápido cada 0.2s
ping -s 1472 8.8.8.8     # paquetes de 1472 bytes (test MTU)

# traceroute — muestra la ruta que toman los paquetes
traceroute 8.8.8.8       # Linux
tracert 8.8.8.8          # Windows
traceroute -n 8.8.8.8    # sin resolver DNS (más rápido)
mtr 8.8.8.8              # traceroute + ping en tiempo real

Leyendo un traceroute

output de ejemplo
# traceroute google.com
 1  192.168.1.1    1.2 ms   ← tu router/gateway
 2  10.10.0.1      5.4 ms   ← router ISP
 3  * * *                   ← hop que no responde ICMP (firewall)
 4  72.14.194.1    15.2 ms  ← red de Google
 5  8.8.8.8        16.8 ms  ← destino

Cada "hop" es un router. El TTL (Time To Live) se decrementa en 1 en cada hop. Cuando llega a 0, el router descarta el paquete y envía un ICMP "Time Exceeded" → traceroute lo captura para mostrar ese hop.

🏴‍☠️ ICMP en Pentesting
  • ICMP Ping Sweep: descubrir hosts activos en una red: nmap -sn 192.168.1.0/24
  • ICMP Tunneling: exfiltrar datos encapsulandolos en paquetes ping (bypassa algunos firewalls). Herramienta: ptunnel
  • Ping of Death: enviar paquetes ICMP malformados para crashear sistemas (histórico, ya no funciona en sistemas modernos)
  • Smurf Attack: DDoS amplificado usando ICMP broadcast dirigido

BGP — Border Gateway Protocol

Internet Routing

¿Qué es BGP?

BGP es el protocolo que hace funcionar Internet. Conecta los Sistemas Autónomos (AS) — los bloques de IPs administrados por ISPs, empresas, y gobiernos. Cada AS anuncia qué prefijos IP (rangos) puede alcanzar. BGP usa el camino con el menor número de ASes para enrutar el tráfico.

🔴 Ataque: BGP Hijacking▶

Un AS malicioso (o comprometido) anuncia rutas falsas para prefijos que no le pertenecen. Si su anuncio es más específico (prefijo /24 vs /16), los routers prefieren la ruta más específica → el tráfico se redirige al atacante.


Casos reales:

  • 2010: China Telecom anunció 50,000 prefijos por 18 minutos — tráfico de US DoD, NASA y Senado fue ruteado por China.
  • 2018: Tráfico de Amazon Route 53 (DNS) fue hijackeado para robar criptomonedas.
  • 2022: BGP hijack para interceptar tráfico de Rostelecom en Rusia.

Defensas:

  • RPKI (Resource Public Key Infrastructure): firma criptográfica de los anuncios BGP. El router valida que el AS tiene derecho a anunciar ese prefijo.
  • BGPsec: extiende RPKI con validación de caminos completos.
  • Filtrado de prefijos: los ISPs configuran prefix-lists para no aceptar anuncios sospechosos.
🔬 Para Profundizar
  • BGP usa TCP puerto 179. Los routers establecen sesiones eBGP entre ASes
  • Herramienta: bgpstream para monitoreo de anuncios BGP en tiempo real
  • Sitios: bgpstream.caida.org, stat.ripe.net
  • Libro recomendado: "BGP Design and Implementation" de Randy Zhang

🏴‍☠️ Transferencia de Zona (AXFR)

Si el nameserver está mal configurado, permite transferencia de zona a cualquier IP — te da todos los registros DNS del dominio: subdominios internos, IPs privadas, servidores de staging, etc. Es como obtener el mapa completo de la infraestructura.

bash
dig AXFR @ns1.vulnerable.com vulnerable.com
# Si está permitido, verás TODOS los registros del dominio
🏴‍☠️ DNS Cache Poisoning

El atacante inyecta respuestas DNS falsas en el caché del resolver. Cuando alguien consulta "banco.cl", el resolver envenenado devuelve la IP del atacante → phishing perfecto. Defensa: DNSSEC valida criptográficamente las respuestas.

🔬 Para Profundizar
  • DNSSEC: firma criptográfica de registros DNS. Valida autenticidad e integridad.
  • DoH (DNS over HTTPS): cifra las consultas DNS en HTTPS — privacidad pero dificulta monitoreo empresarial.
  • DoT (DNS over TLS): cifra en TLS puerto 853 — mismo objetivo que DoH.
  • Subdomain Takeover: CNAME apunta a un servicio eliminado → el atacante registra ese servicio y controla el subdominio.

WHOIS & Reconocimiento Pasivo

OSINT

¿Qué revela WHOIS?

bash
# WHOIS de un dominio
whois google.com

# WHOIS de una IP (quién administra ese bloque)
whois 8.8.8.8

# Salida típica incluye:
# Registrant: empresa/persona que registró el dominio
# Admin Email: email del administrador (útil para phishing)
# Name Servers: ns1.dominio.com, ns2.dominio.com
# Creation/Expiration Date: fechas clave
# Registrar: GoDaddy, Namecheap, etc.
🏴‍☠️ WHOIS en reconocimiento — qué buscar
  • Email del admin: target para spear phishing o credential stuffing
  • Fecha de expiración: si expira pronto y no se renueva → domain hijacking (registras tú el dominio)
  • Name Servers: si usas ns1.empresavulnerable.com, atacar esos NS te da control del DNS
  • Rangos de IP: en WHOIS de IPs obtienes el bloque completo asignado a la empresa → mapa de infraestructura

TLS Handshake — cómo se establece HTTPS

1. Client Hello → versiones TLS soportadas + cipher suites
↓
2. Server Hello → versión elegida + certificado X.509
↓
3. Key Exchange → se acuerda una clave de sesión
↓
4. Finished → todo el tráfico se cifra con la clave de sesión
🔑 Conceptos clave
  • Perfect Forward Secrecy (PFS): cada sesión usa una clave efímera. Si comprometes la clave privada del servidor, NO puedes descifrar sesiones pasadas.
  • TLS 1.3 vs 1.2: TLS 1.3 elimina cipher suites inseguros, usa solo ECDHE para PFS, y reduce el handshake a 1 RTT (más rápido).
  • Certificate Pinning: la app verifica que el certificado coincida con uno específico — previene MITM incluso con CAs comprometidas.
🏴‍☠️ Burp Suite como proxy MITM

Burp Suite actúa como proxy entre tu browser y el servidor. Tu browser confía en el certificado de Burp (lo instalas como CA de confianza), Burp descifra el HTTPS, te permite modificar la request/response, y lo reenvía al servidor real. Es la herramienta #1 para pentesting de aplicaciones web.

configuración básica
# 1. Burp escucha en 127.0.0.1:8080
# 2. Configura browser proxy → 127.0.0.1:8080
# 3. Visita http://burpsuite para instalar CA
# 4. Todo el tráfico HTTPS pasa por Burp descifrado

🔑 POP3 vs IMAP

POP3: descarga y (normalmente) borra del servidor. Un solo dispositivo. Antiguo.
IMAP: sincroniza con el servidor. Múltiples dispositivos ven los mismos emails. Moderno. Si usas Gmail en tu PC y celular, usas IMAP.

🏴‍☠️ Email Spoofing y las defensas DNS

SMTP no requiere autenticación por defecto — cualquiera puede enviar "desde" cualquier dirección. Las defensas son registros TXT en DNS:

  • SPF: lista qué servidores pueden enviar email por tu dominio. Ej: v=spf1 include:sendgrid.net ~all
  • DKIM: firma criptográfica en cada email. El destinatario verifica la firma con la clave pública en DNS.
  • DMARC: política que define qué hacer si SPF/DKIM fallan. Ej: v=DMARC1; p=reject; rua=mailto:dmarc@empresa.com
🏴‍☠️ Herramientas de email en pentesting
  • smtp-user-enum: enumera usuarios válidos en un servidor SMTP vía comandos VRFY/EXPN/RCPT
  • hydra -l user -P wordlist.txt imap://mail.empresa.com: fuerza bruta en IMAP
  • swaks: Swiss Army Knife SMTP — herramienta de prueba de email

bash — gestión de claves SSH
# Generar par de claves (Ed25519 es recomendado en 2024)
ssh-keygen -t ed25519 -C "francisco@empresa.com"
# Crea: ~/.ssh/id_ed25519 (privada) y ~/.ssh/id_ed25519.pub (pública)

# RSA (legacy, compatible con todo)
ssh-keygen -t rsa -b 4096

# Copiar clave pública al servidor
ssh-copy-id user@server
# (la agrega a ~/.ssh/authorized_keys en el servidor)

# Ver huella digital de una clave (para verificar servidor)
ssh-keygen -lf ~/.ssh/id_ed25519.pub

# ssh-agent — no ingresar passphrase cada vez
eval $(ssh-agent)
ssh-add ~/.ssh/id_ed25519
🔑 Ed25519 vs RSA

Ed25519: curva elíptica Curve25519. Claves pequeñas (68 chars), más rápido, más seguro. Recomendado para nuevas claves.
RSA: más compatible (sistemas legacy). Usa mínimo 4096 bits en 2024. Evita 1024/2048 bits.

bash — SSH tunneling
# LOCAL PORT FORWARDING
# Acceder a puerto remoto como si fuera local
# Ej: servicio MySQL en server no expuesto al exterior
ssh -L 3306:localhost:3306 user@server
# Ahora: mysql -h 127.0.0.1 -P 3306 = conecta al MySQL remoto

# REMOTE PORT FORWARDING
# Exponer puerto local a través del servidor remoto
# Útil cuando el cliente está detrás de NAT/firewall
ssh -R 8080:localhost:80 user@server
# Ahora desde server: curl localhost:8080 = tu web local

# DYNAMIC PORT FORWARDING (SOCKS proxy)
# Todo el tráfico del browser pasa por el server SSH
ssh -D 1080 user@server
# Configura browser: SOCKS5 proxy → 127.0.0.1:1080
# Útil en CTFs para acceder a red interna del target
📋 Caso de uso real

Tienes acceso SSH a un servidor en una red corporativa. El servidor de base de datos (PostgreSQL 5432) no está expuesto. Con local forwarding:

bash
ssh -L 5432:db-server:5432 user@bastion
# En otra terminal:
psql -h 127.0.0.1 -U dbuser mydb  # conecta al DB remoto

🔑 Split Tunneling

Configuración donde solo parte del tráfico va por la VPN. Ej: solo el tráfico a la red corporativa usa el túnel, el resto (YouTube, etc.) sale directo. Pros: menos carga en el servidor VPN. Contras: el ISP sigue viendo el tráfico no-tunelizado.

🔑 Rangos de puertos
  • 0–1023: Well-known ports (reservados). Requieren privilegios root para abrirlos.
