Teoría — Prompts, Parsers y Validación

Prompts estructurados para clasificación

Un prompt estructurado para clasificación tiene tres responsabilidades distintas: definir el rol del modelo, especificar exactamente qué categorías existen (y qué significa cada una), e indicar el formato exacto de la salida esperada.

Prompt sin estructura
"Clasifica este texto:\n{texto}"

// Salida posible:
"Esto es una queja del usuario"
// o: "RECLAMO"
// o: "Es un reclamo urgente"
// Inconsistente cada vez
Prompt estructurado
"Clasifica en: consulta|reclamo|urgente|spam|otro.
Responde SOLO con JSON válido:
{'categoria':'...','confianza':0.9,...}"

// Salida: siempre el mismo schema

Output Parsers — qué son y para qué sirven

Un OutputParser convierte el texto que devuelve el LLM en un tipo de dato Python útil. Sin parser, todo es un string. Con parser, obtienes dicts, objetos Pydantic, listas, etc.

StrOutputParser
Solo extrae el texto plano. No transforma nada. Para cuando la respuesta libre es suficiente.
chat básico
JsonOutputParser
Convierte la respuesta del LLM en un dict Python. Puede validar contra un schema Pydantic.
este proyecto
PydanticOutputParser
Convierte y valida contra un modelo Pydantic. Lanza error si el LLM devuelve campos inválidos.
producción
OutputFixingParser
Envuelve otro parser. Si falla, llama al LLM de nuevo pidiéndole que corrija el JSON malformado.
máxima robustez

Validación de salidas con Pydantic

Pydantic es una librería Python que define esquemas de datos con tipos y reglas de validación. Cuando lo combinas con LangChain, el LLM debe producir una salida que pase esa validación — o el parser lanza un error.

Schema Pydantic — ClasificacionTexto
class ClasificacionTexto(BaseModel):
  # Campo 1: categoría con valores permitidos
  categoria: Literal["consulta", "reclamo", "urgente", "spam", "otro"]
  # Campo 2: confianza entre 0.0 y 1.0
  confianza: float # validado: 0.0 ≤ valor ≤ 1.0
  # Campo 3: si debe atenderse pronto
  es_urgente: bool
  # Campo 4: razón de la clasificación
  razon: str # max 150 chars, explicación breve
  # Campo 5: palabras clave detectadas
  palabras_clave: list[str]
Al definir Literal["consulta","reclamo",...], Pydantic rechazará automáticamente cualquier valor que no esté en esa lista. El LLM no puede inventarse una categoría nueva.

Cómo LangChain inyecta el formato en el prompt

El JsonOutputParser con un modelo Pydantic tiene un método .get_format_instructions() que genera automáticamente las instrucciones de formato para incluir en el prompt:

# LangChain genera esto automáticamente:
"Return a JSON object with these fields:
  categoria: string (one of: consulta, reclamo, urgente, spam, otro)
  confianza: float
  es_urgente: boolean
  razon: string
  palabras_clave: array of strings"

Esta instrucción se inyecta en el prompt como variable {format_instructions} — no la escribes a mano.

Genera el JSON según las instrucciones

↓
✅
Validación Pydantic
Parsea JSON → objeto Python tipado
El parser intenta convertir el string JSON en un objeto ClasificacionTexto. Si el LLM devolvió una categoría inválida, un float fuera de rango o campos faltantes, Pydantic lanza un ValidationError. Podemos capturarlo y reintentar.
↓
📤
Objeto Python tipado
ClasificacionTexto con todos los campos validados
El resultado final es un dict (o instancia Pydantic) con categoria, confianza, es_urgente, razon y palabras_clave. Puedes usarlo directamente: if resultado["es_urgente"]: escalar_ticket()

La chain completa en una línea

chain = prompt | model | parser
# prompt formatea el texto + instrucciones
# model genera el JSON
# parser valida y convierte a dict Python

Output Parser en profundidad

Cómo funciona JsonOutputParser

El JsonOutputParser de LangChain hace tres cosas en secuencia cuando recibe la respuesta del LLM:

1
Extrae el JSON del string — busca el bloque JSON aunque el LLM haya agregado texto antes o después (```json ... ``` o texto libre alrededor)
2
Parsea el JSON a dict Python con json.loads()
3
Valida contra el schema Pydantic (si se provee uno) — verifica tipos, valores permitidos y campos requeridos

Por qué temperatura = 0 para clasificación

temperature = 0.7 (creativo)
Clasificación 1: "reclamo"
Clasificación 2: "queja" ← inválido
Clasificación 3: "reclamo_urgente" ← inválido

El modelo "inventa" variantes.
temperature = 0 (determinista)
Clasificación 1: "reclamo"
Clasificación 2: "reclamo"
Clasificación 3: "reclamo"

Mismo texto = mismo resultado siempre.

Estrategia de retry con OutputFixingParser

Si el LLM produce JSON inválido (raro pero posible), OutputFixingParser lo reintenta automáticamente enviando el error al LLM para que se autocorrija:

from langchain.output_parsers import OutputFixingParser

# Envuelve tu parser principal
robust_parser = OutputFixingParser.from_llm(
    parser=parser,
    llm=model
)
# Si el JSON falla, llama al LLM de nuevo con el error
# y le pide que lo corrija — transparente para tu código

Qué pasa cuando falla la validación

OutputParserException: categoria='RECLAMO' no es válido
El LLM devolvió "RECLAMO" (mayúsculas) en lugar de "reclamo" (minúsculas como define el Literal).
Solución: normaliza en el prompt ("usa minúsculas") + OutputFixingParser como red de seguridad.
ValidationError: confianza debe ser float entre 0 y 1
El LLM devolvió confianza: "alta" en lugar de un número.
Solución: en el prompt especifica explícitamente "confianza: número decimal entre 0.0 y 1.0".

Código completo y explicado

1. models/clasificacion.py — el schema

# models/clasificacion.py

from pydantic import BaseModel, Field
from typing import Literal

class ClasificacionTexto(BaseModel):
    categoria: Literal["consulta", "reclamo", "urgente", "spam", "otro"] = Field(
        description="Categoría principal del texto"
    )
    confianza: float = Field(
        ge=0.0, le=1.0,
        description="Nivel de confianza de la clasificación (0.0 a 1.0)"
    )
    es_urgente: bool = Field(
        description="True si requiere atención inmediata"
    )
    razon: str = Field(
        max_length=200,
        description="Breve explicación de la clasificación"
    )
    palabras_clave: list[str] = Field(
        description="Lista de palabras o frases que determinaron la clasificación"
    )
Literal["consulta","reclamo",...]

Restringe el campo a exactamente esos valores. Cualquier otro string fallará la validación de Pydantic.

Field(ge=0.0, le=1.0)

ge = greater or equal, le = less or equal. Pydantic valida que el float esté dentro del rango.

Field(description="...")

La descripción se usa por JsonOutputParser para generar las instrucciones de formato que se inyectan en el prompt.

2. prompts/clasificador.py

# prompts/clasificador.py

from langchain_core.prompts import ChatPromptTemplate

CATEGORIAS = """
- consulta: pregunta o solicitud de información
- reclamo: queja, insatisfacción o reporte de problema
- urgente: requiere atención inmediata (emergencia, amenaza, peligro)
- spam: publicidad no solicitada, mensaje irrelevante o automatizado
- otro: no encaja en ninguna categoría anterior
"""

prompt_clasificador = ChatPromptTemplate.from_messages([
    ("system", """Eres un clasificador experto de textos.
Tu tarea es analizar el texto proporcionado y clasificarlo con precisión.

Categorías disponibles:
{categorias}

REGLAS:
- Usa SOLO las categorías listadas, en minúsculas
- confianza: decimal entre 0.0 y 1.0
- es_urgente: true solo si hay riesgo real o urgencia crítica
- razon: máximo 150 caracteres, sé conciso
- palabras_clave: entre 2 y 5 términos del texto

{format_instructions}"""),
    ("human", "Clasifica este texto:\n\n{texto}"),
])
{categorias}

Variable separada para las definiciones de categorías — así puedes cambiarlas sin editar el prompt base.

{format_instructions}

Esta variable es especial: la genera automáticamente el JsonOutputParser a partir del schema Pydantic. No la escribes tú.

3. clasificador.py — la chain

# clasificador.py

from dotenv import load_dotenv
from langchain_anthropic import ChatAnthropic
from langchain_core.output_parsers import JsonOutputParser
from models.clasificacion import ClasificacionTexto
from prompts.clasificador import prompt_clasificador, CATEGORIAS

load_dotenv()

# ── 1. Modelo con temperatura 0 ────────────────────────
model = ChatAnthropic(
    model="claude-3-5-haiku-20241022",
    temperature=0,  # determinista para clasificación
)

# ── 2. Parser con el schema Pydantic ───────────────────
parser = JsonOutputParser(pydantic_object=ClasificacionTexto)

# ── 3. Construir la chain ───────────────────────────────
chain = prompt_clasificador | model | parser

# ── 4. Función principal ───────────────────────────────
def clasificar(texto: str) -> dict:
    try:
        resultado = chain.invoke({
            "texto": texto,
            "categorias": CATEGORIAS,
            "format_instructions": parser.get_format_instructions(),
        })
        return resultado
    except Exception as e:
        return {
            "categoria": "otro",
            "confianza": 0.0,
            "es_urgente": False,
            "razon": f"Error de clasificación: {str(e)[:100]}",
            "palabras_clave": [],
        }
temperature=0

Para clasificación queremos determinismo. El mismo texto debe dar siempre la misma categoría.

JsonOutputParser(pydantic_object=ClasificacionTexto)

Le pasamos el schema al parser. Así sabe qué validar y puede generar las instrucciones de formato automáticamente.

parser.get_format_instructions()

Este método genera el texto de instrucciones que le dice al LLM exactamente qué JSON debe producir. Se inyecta en el prompt como variable.

except Exception → fallback dict

Si algo falla (JSON malformado, error de red, validación), retornamos un dict válido con valores por defecto. Tu código nunca recibe None.

4. main.py — demo

# main.py

from clasificador import clasificar
import json

def mostrar_resultado(texto: str, resultado: dict):
    cat = resultado["categoria"].upper()
    pct = int(resultado["confianza"] * 100)
    urg = "🔴 URGENTE" if resultado["es_urgente"] else "🟢 Normal"
    print(f"\n{'─'*50}")
    print(f"Texto:      {texto[:60]}...")
    print(f"Categoría:  {cat} ({pct}% confianza)")
    print(f"Urgencia:   {urg}")
    print(f"Razón:      {resultado['razon']}")
    print(f"Palabras:   {', '.join(resultado['palabras_clave'])}")

if __name__ == "__main__":
    textos = [
        "¿Cuál es el horario de atención los fines de semana?",
        "Llevo 3 días sin internet y nadie me soluciona el problema.",
        "GANA $5000 DIARIOS DESDE CASA, HAZ CLIC AQUÍ",
        "Me acaban de robar y necesito ayuda inmediata.",
    ]
    for t in textos:
        mostrar_resultado(t, clasificar(t))

Cómo asegurar salida consistente

La consistencia se logra en capas. Cada capa agrega una garantía adicional.

Capa 1 Prompt restrictivo

Instrucciones claras: "usa SOLO estas categorías", "confianza entre 0.0 y 1.0", "responde ÚNICAMENTE con JSON". Reduce errores del LLM antes de que ocurran.

Capa 2 temperature = 0

Elimina la aleatoriedad. El mismo texto siempre produce la misma clasificación. Crítico para tener reproducibilidad.

Capa 3 format_instructions automáticas

parser.get_format_instructions() genera instrucciones de formato precisas derivadas directamente del schema Pydantic — sin riesgo de que el prompt y el schema queden desincronizados.

Capa 4 Validación Pydantic

Si el LLM devuelve algo que no cumple el schema (tipo incorrecto, valor fuera de rango, campo faltante), Pydantic lanza un error en lugar de dejar pasar datos inválidos.

Capa 5 Fallback dict

El try/except garantiza que la función nunca retorna None ni lanza una excepción no manejada. Siempre retorna un dict con la misma estructura.

Autoevaluación

¿Por qué se usa temperature=0 para clasificación y no 0.7?›
Con temperature=0 el modelo siempre elige el token más probable, haciendo la salida determinista. El mismo texto siempre produce la misma categoría. Con temperature alta, el modelo podría clasificar el mismo texto diferente en dos invocaciones distintas — inaceptable para un clasificador de producción.
¿Qué ventaja tiene Literal["consulta","reclamo",...] vs un campo str libre?›
Con Literal, Pydantic rechaza automáticamente cualquier valor que no esté en la lista. Si el LLM devuelve "RECLAMO", "queja" o "reclamo urgente", la validación falla y puedes manejarlo. Con str libre, cualquier texto pasaría sin error aunque sea incorrecto.
¿Qué hace exactamente parser.get_format_instructions()?›
Genera automáticamente un string de texto con las instrucciones de formato que el LLM debe seguir para producir un JSON válido según el schema Pydantic. Incluye los tipos de datos, valores permitidos y descripciones de cada campo. Se inyecta en el prompt como {format_instructions}.
¿Por qué el try/except en clasificar() retorna un dict en lugar de relanzar el error?›
Porque en producción el código que llama a clasificar() espera siempre un dict con los mismos campos. Si retornáramos None o relanzáramos el error, el código cliente tendría que manejar dos casos. Con el fallback dict, el código cliente es más simple y robusto — siempre recibe la misma estructura.
¿Cuándo usarías OutputFixingParser en lugar de solo JsonOutputParser?›
Cuando la consistencia es crítica y quieres recuperación automática ante errores de formato. OutputFixingParser agrega latencia (hace un segundo llamado al LLM si falla) y costo, así que no siempre conviene. Para prototipo o baja carga: JsonOutputParser + try/except es suficiente. Para producción de alta confiabilidad: OutputFixingParser vale la pena.
