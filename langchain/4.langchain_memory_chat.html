Chatbot con Memoria Conversacional en LangChain

Un chatbot que recuerda todo lo dicho en la sesión. Cada respuesta tiene en cuenta el contexto completo de la conversación anterior.

Python 3.10+ LangChain MessagesPlaceholder InMemoryChatHistory Claude API

Objetivo del proyecto

Construir un chatbot que mantiene el historial de mensajes dentro de la sesión. Cuando el usuario dice "¿y sobre eso que me dijiste antes?", el modelo sabe exactamente a qué se refiere — porque tiene el contexto completo.

Cuándo conviene usar memoria y cuándo no

Usar memoria
✓ Chatbots de soporte o asistentes
✓ Apps donde el usuario hace preguntas relacionadas
✓ Tutores o guías interactivas
✓ Cuando el contexto anterior cambia la respuesta
✓ Entrevistas o formularios conversacionales
No usar memoria
✗ Clasificadores o extractores de datos
✗ Cada llamada es independiente por naturaleza
✗ API con alta concurrencia (memoria = costo)
✗ Tareas de un solo turno (resumir, traducir)
✗ Cuando el historial pasado es irrelevante

Sin memoria vs. con memoria

Esta es la diferencia más importante de entender. Observa lo que pasa en cada turno de la misma conversación:

Sin memoria — cada llamada es nueva
Tú: Me llamo Francisco.
Bot: ¡Hola Francisco!

Tú: ¿Cómo me llamo?
Bot: No sé cómo te llamas, no me has dicho.

Tú: Soy de Chile.
Bot: ¡Interesante!

Tú: ¿De dónde soy?
Bot: No tengo esa información.
Con memoria — historial acumulado
Tú: Me llamo Francisco.
Bot: ¡Hola Francisco!

Tú: ¿Cómo me llamo?
Bot: Te llamas Francisco, me lo dijiste antes.

Tú: Soy de Chile.
Bot: ¡Qué bueno, Francisco de Chile!

Tú: ¿De dónde soy?
Bot: Eres de Chile.

¿Por qué el LLM "olvida" por defecto?

Los LLMs son stateless por diseño. Cada llamada a la API es completamente independiente — el modelo no guarda ningún estado entre requests. Todo el contexto que necesita debe enviarse en cada llamada.

La solución es simple: incluir el historial completo de la conversación en cada llamada. LangChain te da las herramientas para gestionar ese historial automáticamente sin que tengas que hacerlo a mano.

Lo que LangChain envía al LLM con memoria activa

# Lo que ve el LLM en cada llamada (turno 3):
[
  SystemMessage("Eres un asistente útil..."),
  HumanMessage("Me llamo Francisco."),      # turno 1
  AIMessage("¡Hola Francisco!"),            # turno 1
  HumanMessage("Soy de Chile."),            # turno 2
  AIMessage("¡Qué bueno, Francisco!"),      # turno 2
  HumanMessage("¿De dónde soy?"),           # turno 3 ← actual
]

El historial completo viaja en cada request. Eso es la "memoria" — en realidad es contexto acumulado.

Teoría — Memoria, Historial y Contexto

Memoria conversacional

En LangChain, la "memoria conversacional" es el mecanismo que almacena el historial de mensajes y lo inyecta automáticamente en el prompt antes de cada llamada al LLM. No es magia — es gestión estructurada de listas de mensajes.

Historial de mensajes

Un historial es una lista ordenada de objetos mensaje. LangChain define tres tipos de mensaje:

SystemMessage

Instrucciones de comportamiento para el modelo. Generalmente uno por sesión, al inicio.

HumanMessage

Lo que escribió el usuario. Se agrega al historial en cada turno.

AIMessage

La respuesta del modelo. También se guarda en el historial para el próximo turno.

MessagesPlaceholder — la pieza clave

Es un marcador en el ChatPromptTemplate que dice "aquí van los mensajes del historial". LangChain lo reemplaza con la lista real de mensajes al momento de ejecutar la chain.

from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder

prompt = ChatPromptTemplate.from_messages([
    ("system", "Eres un asistente útil."),
    MessagesPlaceholder(variable_name="historial"),  # ← aquí va el historial
    ("human", "{mensaje}"),
])
El orden importa: system primero, luego el historial (para dar contexto), luego el mensaje actual del usuario al final.

Contexto en LLMs — la ventana de contexto

Cada LLM tiene un límite máximo de tokens que puede procesar en una sola llamada (ventana de contexto). El historial consume esos tokens:

Tipos de gestión de memoria en LangChain

LangChain ofrece varias estrategias para gestionar el historial. Entiende cuándo usar cada una:

Historial completo
Guarda y envía todos los mensajes de la sesión. Lo más simple y más fiel — el LLM tiene todo el contexto.
Fácil de implementar
Crece con la conversación
Ventana deslizante
Solo guarda los últimos N turnos. Descarta mensajes antiguos cuando supera el límite.
Costo controlado
Pierde contexto lejano
Resumen de historial
Usa el LLM para resumir la conversación periódicamente y reemplaza el historial por el resumen.
Contexto comprimido
Doble costo de LLM
Memoria semántica
Extrae y almacena hechos clave (nombre, preferencias, etc.) en una base vectorial y los recupera cuando son relevantes.
Muy eficiente en tokens
Complejidad alta
En este proyecto implementamos historial completo (lo más didáctico) con la opción de agregar ventana deslizante como mejora.

RunnableWithMessageHistory — el componente moderno

LangChain moderno (v0.2+) usa RunnableWithMessageHistory para envolver cualquier chain y agregarle gestión automática del historial. Le dices cómo obtener el historial y él se encarga de inyectarlo y actualizarlo en cada llamada.

# En vez de gestionar el historial a mano:
# history.add_user_message(msg)
# response = chain.invoke({"historial": history.messages, ...})
# history.add_ai_message(response)

# Con RunnableWithMessageHistory es automático:
chain_con_memoria = RunnableWithMessageHistory(
    chain,
    get_session_history,  # función que retorna el historial por session_id
    input_messages_key="mensaje",
    history_messages_key="historial",
)

Sesiones múltiples con session_id

Con RunnableWithMessageHistory, diferentes usuarios (o conversaciones) tienen su propio historial separado, identificado por un session_id:

# Usuario A y usuario B tienen historiales separados
chain_con_memoria.invoke(
    {"mensaje": "Hola"},
    config={"configurable": {"session_id": "user_francisco"}}
)
chain_con_memoria.invoke(
    {"mensaje": "Hola"},
    config={"configurable": {"session_id": "user_pedro"}}
)

Código completo y explicado

1. memoria/historial.py

# memoria/historial.py
# Gestiona las sesiones de conversación en memoria

from langchain_community.chat_message_histories import ChatMessageHistory

# Diccionario global que mapea session_id → historial
_sesiones: dict[str, ChatMessageHistory] = {}

def get_session_history(session_id: str) -> ChatMessageHistory:
    """Retorna el historial de una sesión, creándolo si no existe."""
    if session_id not in _sesiones:
        _sesiones[session_id] = ChatMessageHistory()
    return _sesiones[session_id]

def limpiar_sesion(session_id: str) -> None:
    """Borra el historial de una sesión (nuevo chat)."""
    if session_id in _sesiones:
        _sesiones[session_id].clear()

def contar_mensajes(session_id: str) -> int:
    """Retorna cuántos mensajes tiene la sesión."""
    hist = get_session_history(session_id)
    return len(hist.messages)
_sesiones: dict[str, ChatMessageHistory]

Diccionario en memoria que mapea cada session_id a su historial. En producción esto sería Redis o una base de datos.

if session_id not in _sesiones: crear

Patrón "get or create": si la sesión no existe, se crea vacía. Si ya existe, se retorna la existente con todo el historial.

2. chatbot.py — la chain con memoria

# chatbot.py

from dotenv import load_dotenv
from langchain_anthropic import ChatAnthropic
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables.history import RunnableWithMessageHistory
from memoria.historial import get_session_history

load_dotenv()

# ── 1. Modelo ────────────────────────────────────────────
model = ChatAnthropic(
    model="claude-3-5-haiku-20241022",
    temperature=0.7,
)

# ── 2. Prompt con MessagesPlaceholder ────────────────────
prompt = ChatPromptTemplate.from_messages([
    ("system", """Eres un asistente amigable y con buena memoria.
Recuerda todo lo que el usuario te ha contado en esta conversación.
Haz referencias naturales al contexto cuando sea relevante."""),
    MessagesPlaceholder(variable_name="historial"),
    ("human", "{mensaje}"),
])

# ── 3. Chain base ────────────────────────────────────────
chain_base = prompt | model | StrOutputParser()

# ── 4. Envolver con gestión automática de historial ──────
chatbot = RunnableWithMessageHistory(
    chain_base,
    get_session_history,          # cómo obtener/crear el historial
    input_messages_key="mensaje",  # nombre del input del usuario
    history_messages_key="historial", # variable en el prompt
)

# ── 5. Función pública ───────────────────────────────────
def chat(mensaje: str, session_id: str = "default") -> str:
    return chatbot.invoke(
        {"mensaje": mensaje},
        config={"configurable": {"session_id": session_id}},
    )
MessagesPlaceholder(variable_name="historial")

El marcador en el prompt donde se insertará el historial. El nombre debe coincidir exactamente con history_messages_key del RunnableWithMessageHistory.

RunnableWithMessageHistory(chain_base, get_session_history, ...)

Envuelve la chain base con gestión automática de historial. Antes de cada llamada, obtiene el historial e inyecta en el prompt. Después, guarda la respuesta del LLM en el historial.

config={"configurable": {"session_id": ...}}

Así se especifica qué sesión usar. RunnableWithMessageHistory llama a get_session_history(session_id) internamente con este valor.

3. main.py

# main.py

from chatbot import chat
from memoria.historial import limpiar_sesion, contar_mensajes

SESSION_ID = "sesion_principal"

def main():
    print("🧠 Chatbot con Memoria — LangChain")
    print("Comandos: 'limpiar' = nueva sesión | 'salir' = terminar\n")

    while True:
        n = contar_mensajes(SESSION_ID)
        prefijo = f"[{n} msgs] Tú"
        msg = input(f"\n{prefijo}: ").strip()

        if not msg:
            continue
        if msg.lower() == "salir":
            print("¡Hasta luego!")
            break
        if msg.lower() == "limpiar":
            limpiar_sesion(SESSION_ID)
            print("🗑️ Historial borrado. Nueva sesión iniciada.")
            continue

        respuesta = chat(msg, SESSION_ID)
        print(f"Asistente: {respuesta}")

if __name__ == "__main__":
    main()
[{n} msgs] Tú:

El prefijo muestra cuántos mensajes tiene el historial actualmente. Es un recordatorio visual de que la memoria está creciendo.

Limitaciones y cómo mejorar el chatbot

Limitaciones actuales

Memoria solo dura la sesión
Cuando el proceso termina, el historial se pierde. Reiniciar el programa = conversación nueva desde cero.
Solución: persistir el historial en SQLite, Redis o PostgreSQL con los adaptadores de LangChain.
Costo crece con la conversación
Cada mensaje envía todo el historial anterior. Una conversación de 100 turnos envía ~100x más tokens que el primer mensaje.
Solución: ventana deslizante (últimos N mensajes) o resumen periódico del historial.
Un solo usuario / sesión hardcodeada
El código actual usa un SESSION_ID fijo. No escala a múltiples usuarios concurrentes.
Solución: generar session_id dinámico por usuario (uuid.uuid4()) al iniciar cada sesión.
Sin persistencia entre reinicios
ChatMessageHistory vive en memoria RAM — si el servidor cae, se pierde todo.
Solución: SQLChatMessageHistory de langchain-community guarda en SQLite automáticamente.

Cómo mejorar el chatbot

Mejora 1

Persistencia con SQLite

Reemplaza ChatMessageHistory por SQLChatMessageHistory(session_id, connection="sqlite:///chat.db"). Historial que sobrevive reinicios.

Mejora 2

Ventana deslizante

Antes de inyectar el historial, filtra solo los últimos 20 mensajes: history.messages[-20:]. Control del costo sin perder contexto reciente.

Mejora 3

Resumen automático

Cada 30 mensajes, usa el LLM para resumir la conversación y reemplaza los mensajes viejos por el resumen. Contexto comprimido y económico.

Mejora 4

API con FastAPI + sesiones por usuario

Expón el chatbot como POST /chat con {"session_id": "...", "mensaje": "..."}. Cada usuario tiene su propio historial.

Mejora 5

Interfaz Streamlit con historial visible

Usa st.chat_message() de Streamlit para renderizar la conversación con burbuja de chat y mostrar el historial en tiempo real.

Autoevaluación

¿Por qué un LLM "olvida" entre llamadas si no implementas memoria?›
Porque los LLMs son stateless — cada llamada a la API es completamente independiente. El modelo no guarda ningún estado entre requests. Si no envías el historial anterior en cada llamada, el LLM no tiene información sobre lo que se dijo antes.
¿Qué hace exactamente MessagesPlaceholder en el prompt?›
Es un marcador de posición en el ChatPromptTemplate que se reemplaza con la lista real de mensajes del historial al momento de ejecutar la chain. El nombre del placeholder debe coincidir con el history_messages_key del RunnableWithMessageHistory.
¿Cuál es la diferencia entre historial completo y ventana deslizante?›
El historial completo guarda y envía todos los mensajes de la sesión — contexto máximo pero costo creciente. La ventana deslizante solo envía los últimos N mensajes — costo controlado pero pierde contexto lejano. Se elige según la naturaleza del uso: conversaciones cortas → completo, conversaciones largas → ventana.
¿Para qué sirve el session_id en RunnableWithMessageHistory?›
Para identificar qué historial usar. Permite que múltiples usuarios (o conversaciones) tengan sus propios historiales completamente separados. RunnableWithMessageHistory llama a get_session_history(session_id) con ese valor para obtener o crear el historial correspondiente.
¿Cuándo NO conviene usar memoria conversacional?›
Cuando cada llamada es independiente por naturaleza (clasificador, traductor, extractor de datos), cuando el costo es crítico y el historial sería irrelevante, cuando hay alta concurrencia y el historial crecería muy rápido, o cuando la reproducibilidad es importante (el historial puede cambiar el comportamiento del modelo de forma impredecible).
