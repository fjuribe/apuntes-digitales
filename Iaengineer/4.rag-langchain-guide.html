El Ecosistema de Paquetes LangChain

LangChain no es un solo paquete monolítico. Desde 2024 fue separado en módulos especializados para que solo instales lo que necesitas:

⚙️
langchain-core
Interfaces base: Runnable, BaseMessage, BaseLLM. El corazón del framework.
🔗
langchain
Chains, agentes y lógica de orquestación de alto nivel.
🌍
langchain-community
+200 integraciones con el mundo real: loaders, vectorstores, LLMs de terceros.
🤝
langchain-openai / anthropic
Paquetes partner oficiales con soporte directo del proveedor.
💡 ¿Por qué esta separación?

Instalar todo en un solo paquete causaba conflictos de dependencias y builds lentos. Ahora cada integración es opcional.

Instalación correcta

bash
# Instalación base para RAG
pip install langchain langchain-core langchain-community

# Para usar OpenAI como LLM
pip install langchain-openai

# Para Anthropic Claude
pip install langchain-anthropic

# Vector stores populares
pip install chromadb       # ChromaDB (local)
pip install pinecone-client # Pinecone (cloud)
pip install faiss-cpu       # FAISS (local, rápido)

Document Loaders:
Datos del Mundo Real
Los Document Loaders son el primer eslabón de cualquier pipeline RAG. Transforman fuentes externas heterogéneas en el formato estándar Document que LangChain entiende.
📄 PDF, CSV, Web, SQL...
⏱ ~15 min lectura

¿Qué es un Document?

Antes de cargar, entendamos el objeto resultante. Todo Document Loader produce objetos Document:

python
from langchain_core.documents import Document

        "source": "contrato_2024.pdf",

Mini-Proyecto Completo: RAG sobre Drive

python · mini-proyecto-drive.py
import os
from dotenv import load_dotenv
from langchain_community.document_loaders import GoogleDriveLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain_community.vectorstores import Chroma
from langchain.chains import RetrievalQA

load_dotenv()

# 1. Cargar documentos desde Drive
print("📂 Cargando desde Google Drive...")
loader = GoogleDriveLoader(
    folder_id=os.getenv("DRIVE_FOLDER_ID"),
    credentials_path="credentials.json"
)
docs = loader.load()
print(f"✅ {len(docs)} documentos cargados")

# 2. Dividir en chunks
splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000,
    chunk_overlap=200
)
chunks = splitter.split_documents(docs)

# 3. Crear vectorstore
embeddings = OpenAIEmbeddings()
vectorstore = Chroma.from_documents(chunks, embeddings)

# 4. Crear chain RAG
qa_chain = RetrievalQA.from_chain_type(
    llm=ChatOpenAI(model="gpt-4o-mini"),
    retriever=vectorstore.as_retriever(search_kwargs={"k": 4}),
    chain_type="stuff"
)

# 5. Consultar
respuesta = qa_chain.invoke({"query": "¿Cuál es el resumen del documento?"})
print(respuesta["result"])
⚠️ Primera ejecución

Al ejecutar por primera vez, se abrirá el navegador para autorizar la app. Después se guarda el token en token.json para usos futuros.

Text Splitters:
División Inteligente
Los LLMs tienen límites de contexto. Enviar un PDF completo no funciona. Los Text Splitters dividen documentos en chunks manejables sin romper el significado semántico.
✂️ chunk_size, chunk_overlap
⏱ ~14 min

¿Por qué dividir?

📏
Límites de contexto
GPT-4o tiene ~128k tokens. Un PDF de 500 páginas puede superar ese límite fácilmente.
🎯
Precisión en recuperación
Recuperar chunks pequeños y relevantes es más preciso que recuperar documentos enteros.
💰
Costos API
Enviar solo el contexto relevante reduce drásticamente el costo por consulta.

RecursiveCharacterTextSplitter

Es el splitter más recomendado para uso general. Intenta dividir de forma semántica usando una jerarquía de separadores:

Jerarquía de separadores (intenta cada uno en orden):
"\n\n" (párrafo)
→
"\n" (línea)
→
" " (espacio)
→
"" (carácter)
python
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000,      # Tamaño máximo en caracteres
    chunk_overlap=200,    # Solapamiento entre chunks
    length_function=len,   # Función de medición
    add_start_index=True  # Agrega posición en metadata
)

# Con documentos (List[Document])
chunks = splitter.split_documents(docs)

# Con texto puro
texto = "Tu texto largo aquí..."
chunks_texto = splitter.split_text(texto)  # Retorna List[str]

# Inspeccionar resultado
print(f"Chunks generados: {len(chunks)}")
print(f"Primer chunk ({len(chunks[0].page_content)} chars):")
print(chunks[0].page_content[:200])
print(f"\nMetadata: {chunks[0].metadata}")

¿Cómo elegir chunk_size y chunk_overlap?

chunk_size: 1000 chars Recomendado: 500–2000
chunk_overlap: 200 chars Recomendado: 10-20% de chunk_size
Caso de usochunk_sizechunk_overlapPor qué
Contratos legales1500–2000300–400Cláusulas largas con contexto
Artículos técnicos800–1200150–250Secciones moderadas
FAQ / Soporte300–50050–100Preguntas/respuestas cortas
Código fuente500–1000100–200Funciones completas

Otros splitters útiles

python
from langchain.text_splitter import (
    TokenTextSplitter,          # Divide por tokens (más preciso para LLMs)
    MarkdownHeaderTextSplitter, # Divide por headers de Markdown
    PythonCodeTextSplitter,     # Divide código Python por funciones/clases
    HTMLHeaderTextSplitter,     # Divide HTML por encabezados
)

# Por tokens (más preciso que caracteres)
token_splitter = TokenTextSplitter(
    chunk_size=256,   # En TOKENS, no caracteres
    chunk_overlap=50
)

# Por headers de Markdown (ideal para docs técnicos)
headers_to_split = [
    ("#", "H1"),
    ("##", "H2"),
    ("###", "H3"),
]
md_splitter = MarkdownHeaderTextSplitter(
    headers_to_split_on=headers_to_split
)
chunks = md_splitter.split_text(markdown_texto)
¿Para qué sirve el parámetro chunk_overlap en RecursiveCharacterTextSplitter?
A
Para que los chunks sean más pequeños
B
Para que el contexto no se pierda entre chunks contiguos
C
Para evitar chunks duplicados
D
Para mejorar la velocidad de procesamiento
✓ Correcto. El overlap asegura que información que podría estar "en el borde" entre dos chunks aparezca en ambos, preservando el contexto.

Bases de Datos
Vectoriales
Las vector databases almacenan embeddings y permiten búsquedas de similitud ultra-rápidas. Son el motor detrás de la recuperación en RAG.
🗄️ ChromaDB, FAISS, Pinecone
⏱ ~22 min

¿Cómo funciona la búsqueda vectorial?

Query: "¿cláusula rescisión?"
→
Embedding del query
→
Búsqueda ANN
→
Top-K documentos similares

La búsqueda ANN (Approximate Nearest Neighbor) encuentra los vectores más cercanos en el espacio vectorial usando métricas como cosine similarity o distancia euclidiana.

ChromaDB — Vector DB local

python · ChromaDB (recomendado para empezar)
# pip install chromadb
from langchain_community.vectorstores import Chroma
from langchain_openai import OpenAIEmbeddings

embeddings = OpenAIEmbeddings()

# ---- Crear desde documentos ----
vectorstore = Chroma.from_documents(
    documents=chunks,           # List[Document]
    embedding=embeddings,
    persist_directory="./chroma_db",  # Guardar en disco
    collection_name="contratos_legales"
)

# ---- Cargar colección existente ----
vectorstore = Chroma(
    persist_directory="./chroma_db",
    embedding_function=embeddings,
    collection_name="contratos_legales"
)

# ---- Búsqueda de similitud ----
query = "¿Cuáles son las condiciones de rescisión del contrato?"

# Retorna List[Document]
resultados = vectorstore.similarity_search(query, k=4)

# Con scores de similitud (0-1)
resultados_con_score = vectorstore.similarity_search_with_score(query, k=4)

for doc, score in resultados_con_score:
    print(f"Score: {score:.4f} | {doc.page_content[:100]}...")

# ---- Agregar documentos nuevos ----
vectorstore.add_documents(nuevos_chunks)

# ---- Eliminar documentos ----
vectorstore.delete([doc_id_1, doc_id_2])

python · FAISS (Facebook AI)
# pip install faiss-cpu  (o faiss-gpu para GPU)
from langchain_community.vectorstores import FAISS

# Crear
vectorstore = FAISS.from_documents(chunks, embeddings)

# Guardar y cargar (más eficiente que Chroma para datasets grandes)

vectorstore = FAISS.load_local(

# MMR - Maximum Marginal Relevance (diversidad + relevancia)
resultados_mmr = vectorstore.max_marginal_relevance_search(
    query,
    k=4,
    fetch_k=20,   # Candidatos iniciales
    lambda_mult=0.5  # 0=máx diversidad, 1=máx relevancia
)

Retrievers:
Estrategias de Recuperación
Un Retriever es una interfaz que recupera documentos relevantes dada una query. Hay muchas estrategias más allá de la búsqueda vectorial simple.
🔍 MMR, Contextual, BM25...
⏱ ~15 min

Del VectorStore al Retriever

python
# Todo VectorStore puede convertirse en Retriever
retriever = vectorstore.as_retriever(
    search_type="similarity",  # "mmr" | "similarity_score_threshold"
    search_kwargs={
        "k": 4,                # Número de documentos a recuperar
        # Para MMR:
        # "fetch_k": 20,       # Candidatos previos al MMR
        # "lambda_mult": 0.5,  # Balance relevancia/diversidad
        # Para threshold:
        # "score_threshold": 0.8
    }
)

# Usar el retriever directamente
docs = retriever.invoke("¿Qué dice la cláusula 5?")

# Integrar en cualquier chain LCEL
from langchain_core.runnables import RunnablePassthrough

chain = (
    {"context": retriever, "question": RunnablePassthrough()}
    | prompt
    | llm
)

Retrievers Avanzados

🔍 Similarity
Búsqueda por similitud coseno básica. Más rápida.
Default
🎯 MMR
Maximum Marginal Relevance. Balancéa relevancia + diversidad. Evita resultados repetitivos.
Recomendado
📦 ParentDocument
Indexa chunks pequeños pero recupera el documento padre completo. Mejor contexto.
Avanzado
🔢 BM25
Búsqueda lexical clásica. Ideal combinado con vectorial (Hybrid).
Keyword
🗜️ Contextual Compression
Usa un LLM para comprimir los documentos recuperados y extraer solo lo relevante.
Smart
⏰ Time-Weighted
Combina similitud con recencia. Útil para datos que cambian con el tiempo.
Temporal

Contextual Compression Retriever

python · Contextual Compression
from langchain.retrievers import ContextualCompressionRetriever
from langchain.retrievers.document_compressors import LLMChainExtractor

# Paso 1: Retriever base
base_retriever = vectorstore.as_retriever(search_kwargs={"k": 10})

# Paso 2: Compressor que usa un LLM para extraer lo relevante
compressor = LLMChainExtractor.from_llm(llm)

# Paso 3: Combinar
compression_retriever = ContextualCompressionRetriever(
    base_compressor=compressor,
    base_retriever=base_retriever
)

# Ahora recupera documentos ya "comprimidos" y relevantes
docs = compression_retriever.invoke("¿Qué multas aplican por incumplimiento?")
# Cada doc contiene solo el fragmento relevante, no el chunk completo

El MultiQueryRetriever usa un LLM para generar múltiples variaciones de tu query, las ejecuta todas y combina los resultados. Soluciona el problema de que una sola query puede no capturar toda la información relevante.

Úsalo cuando las queries de usuario son cortas o ambiguas, o cuando la terminología en los documentos puede variar (legal, técnico, médico). Tiene un costo adicional en API calls al LLM.

Identificar y cargar la fuente

python · vector_store.py
# vector_store.py
import os
from pathlib import Path
from langchain_community.document_loaders import DirectoryLoader, PyPDFLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_openai import OpenAIEmbeddings
from langchain_community.vectorstores import Chroma

DOCS_PATH = "./documentos/"
CHROMA_PATH = "./chroma_legal_db"

def cargar_y_procesar_documentos():
    """Carga PDFs, divide en chunks y crea el vectorstore."""
    print("📂 Cargando documentos legales...")

    # Cargar todos los PDFs
    loader = DirectoryLoader(
        DOCS_PATH,
        glob="**/*.pdf",
        loader_cls=PyPDFLoader,
        show_progress=True
    )
    docs = loader.load()
    print(f"✅ {len(docs)} páginas cargadas")

    # Dividir en chunks (ajustado para contratos)
    splitter = RecursiveCharacterTextSplitter(
        chunk_size=1500,
        chunk_overlap=300,
        separators=["\n\n", "\n", ".", " "],
        add_start_index=True
    )
    chunks = splitter.split_documents(docs)
    print(f"✂️ {len(chunks)} chunks creados")

    # Crear vectorstore
    print("🔢 Generando embeddings...")
    embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
    vectorstore = Chroma.from_documents(
        documents=chunks,
        embedding=embeddings,
        persist_directory=CHROMA_PATH,
        collection_name="contratos"
    )
    print(f"✅ VectorStore creado: {len(chunks)} vectores")
    return vectorstore

def cargar_vectorstore_existente():
    """Carga un vectorstore ya creado."""
    embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
    return Chroma(
        persist_directory=CHROMA_PATH,
        embedding_function=embeddings,
        collection_name="contratos"
    )

def get_vectorstore():
    if Path(CHROMA_PATH).exists():
        print("📦 Cargando vectorstore existente...")
        return cargar_vectorstore_existente()
    return cargar_y_procesar_documentos()

El formateador convierte los List[Document] del retriever en texto estructurado para el prompt:

python · formateo de documentos
from langchain_core.documents import Document
from typing import List

def format_docs_simple(docs: List[Document]) -> str:
    """Formato básico: solo el contenido."""
    return "\n\n".join(d.page_content for d in docs)

def format_docs_con_fuentes(docs: List[Document]) -> str:
    """Formato con metadatos de fuente (recomendado para legal)."""
    partes = []
    for i, doc in enumerate(docs, 1):
        source = doc.metadata.get('source', 'Desconocido')
        page = doc.metadata.get('page', 'N/A')
        filename = source.split('/')[-1]  # Solo el nombre del archivo

        partes.append(f"""[Documento {i}: {filename}, Página {page}]
{doc.page_content}
{"="*60}""")

    return "\n\n".join(partes)

# [Documento 1: contrato_alquiler.pdf, Página 3]
# El arrendatario se compromete a pagar...
# ============================================================
# [Documento 2: contrato_alquiler.pdf, Página 7]
# En caso de impago, el arrendador podrá...

Chain LCEL completa — rag_chain.py

python · rag_chain.py
# rag_chain.py
from operator import itemgetter
from langchain_openai import ChatOpenAI
from langchain_core.runnables import RunnablePassthrough, RunnableLambda
from langchain_core.output_parsers import StrOutputParser
from langchain.retrievers.multi_query import MultiQueryRetriever
from vector_store import get_vectorstore
from prompts import RAG_PROMPT

def crear_rag_chain(use_multiquery=True):
    """Crea y retorna la chain RAG legal completa."""

    vectorstore = get_vectorstore()
    llm = ChatOpenAI(model="gpt-4o", temperature=0)

    # Retriever base
    base_retriever = vectorstore.as_retriever(
        search_type="mmr",
        search_kwargs={"k": 6, "fetch_k": 20}
    )

    # Opcionalmente usar MultiQuery
    if use_multiquery:
        retriever = MultiQueryRetriever.from_llm(
            retriever=base_retriever,
            llm=ChatOpenAI(model="gpt-4o-mini", temperature=0)
        )
    else:
        retriever = base_retriever

    # Chain LCEL con fuentes
    rag_chain_con_fuentes = (
        RunnablePassthrough.assign(
            context=lambda x: retriever | format_docs_con_fuentes
        )
        | RAG_PROMPT
        | llm
        | StrOutputParser()
    )

    # Chain alternativa que también retorna los docs fuente
    rag_chain_completo = RunnablePassthrough.assign(
        source_documents=itemgetter("question") | retriever,
        answer=(
            RunnablePassthrough.assign(
                context=lambda x: format_docs_con_fuentes(
                    retriever.invoke(x["question"])
                )
            )
            | RAG_PROMPT | llm | StrOutputParser()
        )
    )

    return rag_chain_con_fuentes

# Uso:
chain = crear_rag_chain()
respuesta = chain.invoke("¿Hay cláusula de confidencialidad?")
print(respuesta)

# Streaming:
for chunk in chain.stream("¿Cuáles son las penalidades por incumplimiento?"):
    print(chunk, end="", flush=True)

Combina búsqueda vectorial (semántica) con BM25 (lexical) para mejores resultados:

python · hybrid_retriever.py
# pip install rank-bm25 langchain-community
from langchain_community.retrievers import BM25Retriever
from langchain.retrievers import EnsembleRetriever

def crear_retriever_hibrido(chunks, vectorstore):
    """
    Combina búsqueda vectorial + BM25 (keyword).
    Ideal para: nombres propios, artículos legales específicos,
    fechas exactas, que los embeddings pueden perder.
    """
    # Retriever 1: Vectorial (semántico)
    vector_retriever = vectorstore.as_retriever(
        search_type="mmr",
        search_kwargs={"k": 5}
    )

    # Retriever 2: BM25 (keyword/lexical)
    bm25_retriever = BM25Retriever.from_documents(chunks)
    bm25_retriever.k = 5

    # Ensemble: combina ambos con pesos
    hybrid_retriever = EnsembleRetriever(
        retrievers=[bm25_retriever, vector_retriever],
        weights=[0.4, 0.6]  # 40% BM25, 60% vectorial
    )

    return hybrid_retriever

# Uso en la chain:
hybrid = crear_retriever_hibrido(chunks, vectorstore)

rag_hibrida = (
    {
        "context": hybrid | format_docs_con_fuentes,
        "question": RunnablePassthrough()
    }
    | RAG_PROMPT | llm | StrOutputParser()
)

# Funciona mejor para queries con términos exactos:
respuesta = rag_hibrida.invoke("Artículo 7, cláusula de indemnización")
