¿Qué es un Retriever?

Un retriever recupera documentos relevantes dado un query del usuario. Es la pieza que diferencia un LLM genérico de uno que "sabe" sobre tus datos. En LangChain todos implementan .get_relevant_documents(query), lo que los hace intercambiables en cualquier cadena LCEL.

🔍

Entrada

Un string con la pregunta del usuario. Nada más.

📄

Salida

Lista de Document con texto recuperado y metadatos.

🔗

En LCEL

Se conecta con | como cualquier componente de la cadena.

🔄

Intercambiable

Cambiar de retriever = cambiar una línea. Tu cadena no se entera.

Básicos — El punto de partida

Sin dependencias extra. Funcionan desde el primer día con cualquier vector store.

🗄️
VectorStore Retriever
básico

El más fundamental. Convierte tu query en un embedding y busca los N documentos más similares por distancia coseno en un vector store (FAISS, Chroma, Pinecone…).

¿Cuándo usarlo?

Punto de partida para cualquier RAG. Funciona bien para documentación técnica, FAQs y bases de conocimiento donde la similitud semántica es suficiente.

retriever = vectorstore.as_retriever( search_type="similarity", search_kwargs={"k": 4} ) # En una cadena LCEL chain = retriever | format_docs | prompt | llm
Complejidad
🌈
MMR — Max Marginal Relevance
básico

Variante del VectorStore que balancea relevancia con diversidad. Evita que todos los documentos recuperados digan lo mismo. Controla el balance con lambda_mult.

¿Cuándo usarlo?

Cuando tus datos tienen mucha redundancia (varios docs sobre el mismo tema) y quieres que el LLM reciba perspectivas variadas.

retriever = vectorstore.as_retriever( search_type="mmr", search_kwargs={ "k": 4, "fetch_k": 20, # candidatos iniciales "lambda_mult": 0.5 # 0=máx diversidad } )
Complejidad
📊
BM25 Retriever
básicohíbrido

Búsqueda léxica clásica (TF-IDF mejorado). No usa embeddings — busca documentos que contengan las palabras exactas del query. Rápido y predecible.

¿Cuándo usarlo?

Cuando el usuario busca términos técnicos, nombres propios, códigos o IDs exactos. También ideal como complemento en búsqueda híbrida.

from langchain_community.retrievers import BM25Retriever retriever = BM25Retriever.from_documents( docs, k=4 ) # pip install rank_bm25
Complejidad
🔵

Híbridos — Lo mejor de dos mundos

Combinan búsqueda semántica y léxica. El estándar para producción.

⚡
Ensemble Retriever
híbrido

Combina múltiples retrievers usando Reciprocal Rank Fusion (RRF). Une la potencia semántica del VectorStore con la precisión léxica del BM25.

¿Cuándo usarlo?

En producción con datasets heterogéneos. El combo más recomendado: BM25 + VectorStore con pesos 40/60. Mejora la cobertura sin overhead de LLM adicional.

from langchain.retrievers import EnsembleRetriever ensemble = EnsembleRetriever( retrievers=[bm25_retriever, vector_retriever], weights=[0.4, 0.6] # BM25 40%, vector 60% ) # Devuelve docs rankeados por RRF
Complejidad
💡 Combo ganador en producción Ensemble(BM25 + VectorStore) con pesos 40/60 es el setup más recomendado. Cubre búsquedas léxicas (términos exactos) y semánticas (significado) sin añadir llamadas al LLM ni coste extra de tokens.
🟣

Con IA — Usan un LLM internamente

Más potentes, pero consumen tokens adicionales por cada query.

🧠
MultiQuery Retriever
con ia

Usa un LLM para generar múltiples variantes del query original, ejecuta la búsqueda con cada una y une los resultados. Ataca el problema desde ángulos distintos.

¿Cuándo usarlo?

Cuando los usuarios escriben queries ambiguos o vagos. Ideal para chatbots donde el usuario no sabe exactamente cómo preguntar.

from langchain.retrievers.multi_query import MultiQueryRetriever retriever = MultiQueryRetriever.from_llm( retriever=base_retriever, llm=chat_model ) # Genera 3 queries, hace 3 búsquedas, # une y deduplica resultados
Complejidad
✂️
Contextual Compression
con ia

Usa un LLM para comprimir y filtrar los documentos recuperados, extrayendo solo los fragmentos relevantes al query. Reduce el ruido antes de llegar al LLM principal.

¿Cuándo usarlo?

Cuando tus chunks son grandes y contienen información mezclada. Mejora la precisión de la respuesta y ahorra tokens de contexto.

from langchain.retrievers import ContextualCompressionRetriever from langchain.retrievers.document_compressors import LLMChainExtractor compressor = LLMChainExtractor.from_llm(llm) retriever = ContextualCompressionRetriever( base_compressor=compressor, base_retriever=base_retriever )
Complejidad
🎯
Self-Query Retriever
con iaavanzado

Un LLM traduce el query en lenguaje natural a filtros de metadatos estructurados. Permite búsquedas como "docs de 2023 sobre seguridad" de forma automática.

¿Cuándo usarlo?

Cuando tus documentos tienen metadatos ricos (fecha, autor, categoría) y los usuarios necesitan filtrar sin saberlo explícitamente.

from langchain.retrievers.self_query.base import SelfQueryRetriever retriever = SelfQueryRetriever.from_llm( llm=llm, vectorstore=vectorstore, document_contents="Políticas de empresa", metadata_field_info=metadata_info ) # "políticas de 2024 de RRHH" # → filter: {year:2024, dept:"RRHH"}
Complejidad
🟠

Avanzados — Para casos de uso específicos

Resuelven problemas concretos de arquitectura. Sin overhead de LLM, pero requieren más configuración.

🌳
ParentDocument Retriever
avanzado

Indexa chunks pequeños para mejor búsqueda, pero devuelve el documento padre completo para mejor contexto. Soluciona el trade-off clásico del chunking.

¿Cuándo usarlo?

En documentos largos donde fragmentos pequeños son mejores para encontrar, pero el LLM necesita el contexto amplio para responder bien.

from langchain.retrievers import ParentDocumentRetriever from langchain.storage import InMemoryStore retriever = ParentDocumentRetriever( vectorstore=vectorstore, docstore=InMemoryStore(), child_splitter=small_splitter, # 200 chars parent_splitter=large_splitter # 2000 chars )
Complejidad
💡 El truco del chunking ParentDocument resuelve el dilema eterno: chunks pequeños (200 chars) para indexar y encontrar bien, chunks grandes (2000 chars) para que el LLM tenga contexto suficiente para responder. No elijas uno — usa ambos a la vez.

Cheat Sheet — Lo que nunca cambia

Todo retriever implementa .get_relevant_documents(query). Son intercambiables en cualquier cadena LCEL sin cambiar el resto del código.

chain = retriever | format_docs | prompt | llm. El retriever siempre va primero. Su output es List[Document].

⚡
