01
Imports — Las piezas de LEGO
siempre igual ▼

Cada chatbot con LangChain importa 3 familias de cosas. Esto es constante — lo que cambia es solo cuál proveedor usas (OpenAI, Google, Anthropic…). 

# ─── FAMILIA 1: La interfaz visual ───
import streamlit as st

# ─── FAMILIA 2: El cerebro (el modelo LLM) ───
from langchain_google_genai import ChatGoogleGenerativeAI
# ↑ Si usaras OpenAI sería: from langchain_openai import ChatOpenAI
# ↑ Si usaras Anthropic: from langchain_anthropic import ChatAnthropic

# ─── FAMILIA 3: Las herramientas de LangChain ───
from langchain.schema import AIMessage, HumanMessage, SystemMessage
from langchain.prompts import PromptTemplate
🔁 Patrón constante
Siempre importas: (1) tu framework de UI, (2) un modelo Chat de LangChain, (3) los tipos de mensaje AIMessage/HumanMessage y PromptTemplate. Solo cambia el proveedor.
💡 ¿Por qué AIMessage y HumanMessage?
LangChain envuelve cada mensaje en un "tipo" para que el LLM sepa quién dijo qué. Es como las etiquetas "Tú:" y "Bot:" pero estandarizadas. Esto es obligatorio para manejar el historial de conversación.
02
Configuración de la UI — La cáscara visual
config ▼

Streamlit te da componentes listos para armar la interfaz. Estos 3 métodos los vas a usar en casi todos tus proyectos: 

# ─── Metadata de la página (siempre al inicio) ───
st.set_page_config(page_title="Chatbot", page_icon="🤖")
st.title("🤖 Chatbot con LangChain")

# ─── Sidebar = Panel de control ───
with st.sidebar:
    temperature = st.slider("Temperatura", 0.0, 1.0, 0.5)
    model_name  = st.selectbox("Modelo", ["gemini-2.5-flash"])
💡 ¿Qué es la "Temperatura"?
Va de 0.0 a 1.0. 0 = respuestas predecibles (para código, datos exactos). 1 = respuestas creativas (para poesía, brainstorming). Es el "nivel de creatividad" del bot.
🔁 Patrón constante
st.set_page_config() siempre va primero. El st.sidebar es donde pones los controles que no quieres en el chat principal. Esto es estándar en toda app Streamlit.
03
Inicialización del Modelo — El cerebro
siempre igual ▼

Aquí creas la conexión con el LLM. Siempre son los mismos 2 parámetros mínimos: cuál modelo y qué temperatura. Lo único que cambia es la clase. 

# ─── Google Gemini ───
chat_model = ChatGoogleGenerativeAI(
    model=model_name,
    temperature=temperature,
    google_api_key=GOOGLE_API_KEY
)

# ─── Si fuera OpenAI, sería EXACTAMENTE igual: ───
# chat_model = ChatOpenAI(model="gpt-4", temperature=0.7)

# ─── Si fuera Anthropic: ───
# chat_model = ChatAnthropic(model="claude-sonnet-4-20250514", temperature=0.7)
🔁 Patrón constante
La fórmula es siempre: modelo = ChatXXX(model=..., temperature=..., api_key=...). LangChain estandariza TODOS los proveedores con la misma interfaz. Cambiar de proveedor = cambiar 1 línea.
💡 ¿Por qué esto es genial?
LangChain abstrae al proveedor. Tu código de chat, prompts, cadenas, etc. funciona idéntico sin importar si usas Google, OpenAI o Anthropic. Solo cambias el import y la línea del modelo.
04
Session State — La memoria del chat
siempre igual ▼

Streamlit re-ejecuta TODO el script cada vez que el usuario interactúa. Sin session_state, los mensajes se perderían. Este bloque es obligatorio y siempre idéntico. 

# ─── Crear la "memoria" si no existe ───
if "mensajes" not in st.session_state:
    st.session_state.mensajes = []

# ─── Renderizar mensajes guardados ───
for msg in st.session_state.mensajes:
    role = "assistant" if isinstance(msg, AIMessage) else "user"
    with st.chat_message(role):
        st.markdown(msg.content)
🔁 Patrón constante
Siempre tendrás estas 2 partes:
① if "X" not in st.session_state → inicializar
② for msg in st.session_state.X → re-dibujar
Esto es el "contrato" de Streamlit para persistir datos entre reruns.
💡 ¿Por qué isinstance(msg, AIMessage)?
Recuerda que guardamos los mensajes como HumanMessage y AIMessage. Con isinstance() preguntamos "¿este mensaje es del bot?" para elegir el avatar correcto en la UI.
05
Prompt + Cadena (LCEL) — El alma del bot
lógica core ▼

Aquí defines qué personalidad tiene tu bot y cómo conectas el prompt con el modelo. El operador | (pipe) es la magia de LCEL. 

# ─── PASO 1: Define el template del prompt ───
prompt_template = PromptTemplate(
    input_variables=["mensaje", "historial"],
    template="""Eres un asistente útil llamado ChatBot Pro.
Historial: {historial}
Pregunta: {mensaje}"""
)

# ─── PASO 2: Conecta prompt → modelo con el pipe ───
cadena = prompt_template | chat_model

# Así de simple. Esto es LCEL.
# Lee como: "toma el prompt, pásalo al modelo"
🔁 Patrón constante
La fórmula LCEL siempre es: cadena = prompt | modelo. Puedes extenderla: cadena = prompt | modelo | parser, pero la base es siempre esa. El | es como un "tubo" que pasa la salida de uno como entrada del siguiente.
💡 ¿Por qué {historial} en el prompt?
Sin esto, el bot no recuerda lo que dijiste antes. Al inyectar el historial en cada mensaje nuevo, le das "contexto" al modelo. Es una técnica manual — existen alternativas más avanzadas como ConversationBufferMemory, pero esta es la más clara para empezar.
06
El Loop del Chat — Donde todo cobra vida
ui + lógica ▼

Este es el ciclo de interacción: usuario escribe → se muestra → bot responde con streaming → se guarda. Esto es esencialmente el mismo patrón en todos los chatbots Streamlit. 

# ─── Input del usuario ───
pregunta = st.chat_input("Escribe tu mensaje:")

if pregunta:
    # 1️⃣ Mostrar lo que escribió el usuario
    with st.chat_message("user"):
        st.markdown(pregunta)

    # 2️⃣ Generar respuesta con STREAMING
    try:
        with st.chat_message("assistant"):
            placeholder = st.empty()
            full_response = ""
            
            for chunk in cadena.stream({
                "mensaje": pregunta,
                "historial": st.session_state.mensajes
            }):
                full_response += chunk.content
                placeholder.markdown(full_response + "▌")
            placeholder.markdown(full_response)

        # 3️⃣ Guardar ambos mensajes en la memoria
        st.session_state.mensajes.append(HumanMessage(content=pregunta))
        st.session_state.mensajes.append(AIMessage(content=full_response))

    except Exception as e:
        st.error(f"Error: {e}")
🔁 Patrón constante
El loop siempre tiene estos 3 pasos:
① Mostrar el mensaje del usuario → st.chat_message("user")
② Generar respuesta → cadena.stream() o cadena.invoke()
③ Guardar ambos → session_state.mensajes.append()
💡 ¿Qué hace .stream() vs .invoke()?
.invoke() = espera toda la respuesta y la muestra de golpe.
.stream() = va mostrando palabra por palabra (como ChatGPT). El truco del "▌" al final simula el cursor parpadeante. Es solo estética, pero da una experiencia profesional.
💡 ¿Por qué try/except?
Las APIs fallan: límites de tasa, API keys inválidas, errores de red. El try/except evita que tu app explote y muestra un mensaje amigable al usuario. Siempre envuelve las llamadas al LLM en try/except.

Los 3 imports

Crear modelo

ChatXXX(model=..., temperature=..., api_key=...) — misma firma para todos los proveedores

LCEL Pipe

Session State

if "X" not in st.session_state → inicializar. Obligatorio porque Streamlit re-ejecuta todo el script.

El loop de chat

Mostrar → Generar (.stream()) → Guardar (.append()). Siempre en ese orden.

Try/Except

Siempre envuelve la llamada al LLM. Las APIs fallan, y tu app no debería explotar por eso.
